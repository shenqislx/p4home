# project-milestones Plan

## 1. 背景

在进入真实功能开发前，需要先为项目建立统一的里程碑体系，便于按阶段检查开发进度、评估风险，并判断是否具备进入下一阶段的条件。

## 2. 目标

- 为 `p4home` 定义清晰的阶段性 milestones
- 每个 milestone 具备明确的目标、交付物、验收标准和风险门槛
- 让后续功能开发可以映射到统一的进度坐标系

## 3. 范围

包含：

- 项目 0 到 1 的阶段划分
- 每阶段的交付物
- 每阶段的完成定义
- 建议的阶段审查方式

不包含：

- 具体某个功能模块的实现细节
- 具体某一次提交的任务拆分

## 4. 设计方案

### 4.1 目录影响

- 新增 `docs/project-milestones.md`

### 4.2 模块拆解

- 里程碑主文档
- 阶段验收标准
- 进度检查建议

### 4.3 数据流 / 控制流

- 新功能开发前先参考 milestones 确认所处阶段
- 每个功能 plan 要标明它属于哪个 milestone
- milestone 的完成以“验收标准满足”为准，而不是以“代码量”为准

## 5. 实现任务

1. 创建里程碑 plan
2. 设计项目阶段划分
3. 编写 milestones 文档
4. 自查内容与现有方案文档是否一致

## 6. 测试方案

### 6.1 构建验证

- 不适用

### 6.2 功能验证

- 检查 milestones 是否覆盖项目主线
- 检查每个 milestone 是否有可执行的验收标准
- 检查是否方便用户据此检查进度

### 6.3 回归验证

- 与现有技术路线保持一致：`ESP32-P4 原生面板`、`ESP-IDF + LVGL + ESP-SR`
- 不与现有 harness 流程冲突

### 6.4 硬件/联调验证

- 不适用

## 7. 风险

- 阶段划分过粗会导致进度不可见
- 阶段划分过细会导致维护成本过高
- 验收标准若不够明确，后续容易产生“完成但不可交付”的情况

## 8. 完成定义

- `docs/project-milestones.md` 创建完成
- 每个 milestone 含目标、交付物、验收标准、退出条件
- 可直接用于后续功能计划引用

## 9. review 准备

- 待用户重点确认里程碑的粒度是否合适
- 待用户确认是否需要把 milestone 映射到具体时间周期
